# Test Review #1 — `tests/`

- **Date:** 2026-05-31
- **Reviewer:** Claude (automated)
- **Scope:** `tests/` directory at `HEAD` — 14 test modules, ~2,250 LoC of test code
- **Categories used:**
  - **Brittle** — passes today but is fragile to a refactor, depends on private state, time, or RNG state in ways that can break silently.
  - **Incomplete / no-op** — the test runs but doesn't actually assert what its name claims, or only proves the test scaffolding works.
  - **Improvable** — solid as-is, but small changes would raise its value.
  - **Missing scenarios** — important behaviors of the SUT that no test currently covers.

## 0. Executive Summary

The suite is impressively complete for a v1 codebase — most modules have both unit-level fakeredis tests and a real-container e2e flow. There are no clearly-useless tests, and assertions generally match function names.

The largest gaps are:

| Theme | Files affected | Severity |
|---|---|---|
| Concurrency tests run on fakeredis serially → don't actually test concurrency | `test_redis_state.py`, `test_breaker.py` | High |
| Brittle reliance on RNG state / private buffers / wall-clock sleeps | `test_router.py`, `test_accounting.py` | High |
| `test_app_e2e.py` references env vars/features the production code doesn't implement (`GATEWAY_SEED_CALLERS`); mutates `os.environ` at module scope | `test_app_e2e.py` | High |
| No test for `estimate_tokens` (a pure function that drives bucket sizing) | `test_ratelimit.py` | Medium |
| No test for `/v1/usage` (IDOR endpoint flagged in cr-1) or for `/readyz` | `test_app_e2e.py` | Medium |
| Mock-vendor content-filter mapping untested for `openai.finish_reason=='content_filter'` and `google` SAFETY | `tests/providers/*` | Medium |
| `Message.content` size, `max_tokens` upper bound, and `metadata` schema have no negative tests because the code lacks the validators (matches cr-1 4.1) | `test_models.py` | Medium |

## 1. `tests/conftest.py`

```python
@pytest_asyncio.fixture
async def redis():
    r = fakeredis_aio.FakeRedis(decode_responses=False)
    try:
        yield r
    finally:
        await r.flushall()
        await r.aclose()
```

**Improvable**

- This is the only shared fixture. Every other test file rolls its own clock-injection (`clock = [0.0]`, `now_s = lambda: clock[0]`), its own `_config()` builder, its own `_caller()`, and its own DSN strip. There's enough duplication across `test_router.py`, `test_refresh.py`, `test_app_e2e.py`, and `test_db.py` that hoisting builders into `conftest.py` would prevent inevitable drift (already see slight differences in `routing.min_weight_floor` across files).
- No `freeze_time` / `monotonic_clock` fixture, so each test reinvents fake-clock plumbing.
- No autouse fixture to **reset `prometheus_client.REGISTRY`** between tests. The gauges and counters in `gateway/metrics.py` live at module scope; running the e2e tests in the same process leaks state into any later test that scrapes `/metrics` (see `test_app_e2e.py:test_weighted_distribution_approximates_config`).

**Missing scenarios**

- A shared `pg_dsn` / `redis_url` fixture for testcontainers so `test_db.py` and `test_app_e2e.py` don't each spin up their own Postgres (saves ~10s/run).
- A `cfg_builder` fixture parameterized by tier / pricing / rate-limit.

## 2. `tests/test_accounting.py`

### 2.1 `test_enqueue_under_capacity_no_drops`
**Improvable.** Asserts `len(flat) == 5` but doesn't assert *which* records were written — a writer that wrote unrelated records would pass. Compare `request_id`s to the input set.

### 2.2 `test_drops_oldest_when_full`
**Brittle.**
```python
assert len(q._buffer) == 3
ids = [r.request_id for r in q._buffer]
```
Reads private `_buffer` and `_dropped_total`. A refactor to `collections.deque(maxlen=capacity)` (which is the natural simplification — see cr-1 6.x note) would change `_buffer` semantics and break this test.

**Fix:** expose `len(queue)` or a `peek_ids()` helper; assert via the public API. Alternatively, drive the queue at `flush_size=1` so writes are visible through `_FakeWriter`.

### 2.3 `test_flushes_at_size_threshold`
**Brittle.** Relies on `asyncio.sleep(0.05)` after enqueueing 5 records, hoping the drain task gets scheduled. On a loaded CI it's not guaranteed. The assertion `any(len(b) == 5 for b in writer.batches)` permits multiple batches, so a single 5-record flush triggered by the *stop* (not by the size threshold) would also pass — meaning the test doesn't actually distinguish "size threshold worked" from "stop drained".

**Fix:** snapshot `writer.batches` *before* calling `stop()`; assert exactly one batch with 5 records existed before shutdown.

### 2.4 `test_flushes_at_time_threshold`
**Brittle.** Same sleep-based timing assumption (`flush_interval_ms=30`, `asyncio.sleep(0.1)`). Also passes if the queue did nothing and the final stop-drain caught the record. Combine with 2.3 by introducing a `flush_event: asyncio.Event` that fires on each successful flush and assert it fires before stop.

### 2.5 `test_stop_drains_remaining`
**Solid.**

### Missing scenarios
- Writer raises mid-flush → records counted in `dropped_total` and not re-enqueued. (`_flush`'s `except Exception:` branch is uncovered.)
- `enqueue` from many concurrent tasks (gather over 1000 enqueues) — order preserved? drop accounting accurate?
- `start()` called twice (idempotent? current code is).
- `stop()` called before `start()` (currently sets stop event and tries to await `None` — would AttributeError? actually `if self._task:` guards it; worth a test).
- `dropped_total` exposed live on metrics (cr-1 §6.3 — currently only on shutdown).
- Slow writer (sleep) backs queue up to capacity → asserts oldest is evicted.

### 4.5 `test_only_one_concurrent_probe_holder`
**Brittle — passes for the wrong reason.**
```python
results = await asyncio.gather(
    *[b.try_probe("openai", "gpt-4o") for _ in range(6)]
)
assert sum(1 for r in results if r) == 1
```
`fakeredis` runs in-process and processes each command serially on the event loop. The six `gather`-ed coroutines effectively run sequentially. This test passes because `SET NX` is sequentially atomic — but it would also pass even if the implementation had a race that real Redis would expose. **Fix:** add a comment explaining the limitation; cover real contention in `test_app_e2e.py` against the testcontainers Redis with `asyncio.gather` over `try_probe`.

- **HALF_OPEN → OPEN → HALF_OPEN cycle**: two consecutive open windows; assert `opened_at_s` is updated on the second open.
- **`state()` for never-seen-candidate** returns CLOSED (covered by `test_starts_closed` but the SCAN seed path is what runs).
- **`try_probe` lock TTL**: holder expires after `ttl_s`; next caller can acquire. Currently no test exercises the 10s TTL.
- **Pub/sub** broadcast of state transitions — the module docstring mentions it ("pub/sub publishes transitions") but neither the code nor the tests implement it. Either remove the docstring claim or add the feature + test.
- **Concurrent record_success during refresh_snapshot** — `BreakerSet._snapshot` is mutated from multiple paths (cr-1 §6.1).
- **`_compute_next_state` floating-point edge** when `failures / total == failure_threshold` exactly.

## 9. `tests/test_router.py`

### 9.1 `harness` fixture
**Improvable.** Returns a dict of 11 keys; tests dig into it with string keys. A small dataclass or NamedTuple would type-check and self-document.

### 9.2 `test_success_first_attempt`
**Solid.** Could additionally assert `result.attempts[0].latency_ms >= 0` and `vendor_req_id` propagated to the response id.

### 9.3 `test_failover_after_rate_limited`
**Brittle — RNG-peek pattern.**
```python
rng_peek = random.Random(123)
first_pick = engine.pick(cfg.tiers["fast"], exclude=set(), rng=rng_peek)
...
harness["vendors"][first_pick.provider].queue_error(RateLimited("429"))
```
`engine.pick` currently consumes the RNG state in one `rng.random()` call, so a parallel `Random(123)` aligns. If `pick()` ever changes to use `rng.choices(...)` or calls `random()` twice, the peek diverges silently and the wrong vendor gets the queued error — the test would then pass for a different reason (or fail confusingly).

**Fix options (in order of robustness):**
- Queue the `RateLimited` error on *every* vendor; expect the first attempt to fail and the second to succeed. Remove the peek.
- Or, replace `random.Random(123)` with a deterministic mock that returns 0.0 (always picks first), and queue the error on the highest-base-weight candidate.

### 9.4 `test_all_candidates_fail_returns_503`
**Solid.** Queues 3 errors per vendor (enough for the pool) and asserts `UPSTREAM_UNAVAILABLE`.

### 9.5 `test_bad_request_returns_immediately`
**Solid.** Asserts `total_calls == 1`, proving no failover happens.

### 9.6 `test_deadline_exceeded_mid_failover`
**Brittle and weak.**
```python
assert exc.value.kind in (
    RouterErrorKind.DEADLINE_EXCEEDED,
    RouterErrorKind.UPSTREAM_UNAVAILABLE,
)
```
The assertion accepts either error kind — meaning the test does NOT actually pin down deadline-exceeded behavior. Also re-assigns `v.chat = chat_eat_budget` after queueing errors, which makes the control flow hard to reason about. The "clock-advancing vendor" defined as `_ClockAdvancingVendor` is dead code (never assigned to any vendor).

**Fix:**
- Use a single vendor that adds `9.5s` to the deadline clock and raises `Transient5xx`.
- Assert `exc.value.kind is RouterErrorKind.DEADLINE_EXCEEDED` strictly.
- Assert `len(attempts) >= 1` so we know failover started before the budget ran out.

### 9.7 `test_breaker_open_candidate_not_picked`
**Improvable.** Loops 30 times — passes today because seeded RNG never picks the (now zero-weighted) breaker-open candidate. Could be tightened by asserting the engine's `_weight()` for `openai/gpt-4o-mini` returns 0 directly (no need for the 30-iteration randomized confirmation).

### 9.8 `test_empty_bucket_causes_repick`
**Weak / partially incomplete.**
```python
for _ in range(20):
    r = await router.route(_req(), _caller())
    assert r.response.choices[0].message.role == "assistant"
```
The assertion only checks "got a successful response" — it does NOT verify that anthropic was actually excluded after bucket exhaustion. A bug where the bucket drain didn't matter and anthropic was still served would pass this test.

**Fix:** assert `r.attempts[0].provider != "anthropic"` (or examine `attempts` for `bucket_empty`).

### 9.9 `test_failed_attempt_recorded_in_observe`
**Improvable.** Same RNG-peek as 9.3, same fix. Also: only verifies `failures == 1` for the failed candidate; doesn't verify the *successful* candidate's `successes == 1`.

### Missing scenarios
- **AuthError propagates as 401** (router translates to `RouterErrorKind.AUTH`); currently only `INVALID_REQUEST` is tested.
- **ContentFiltered propagates as 400** with `type="content_filtered"`.
- **Vendor missing path** (`vendors.get(cand.provider) is None`) — branch at `router.py:181-185` is uncovered.
- **Per-attempt timeout shrinks near deadline** — `attempt_timeout = min(max(0.1, remaining - buffer_s), per_attempt_max_s)`; assert that with 1.5s remaining and 0.5s buffer the per-attempt timeout is 1.0s, not 8.0s.

## 11. `tests/test_app_e2e.py`

### 11.1 `stack` fixture
**Brittle — multiple issues.**

a) Mutates global `os.environ` at *module scope* without `monkeypatch`. Any later test in the same pytest session reads these env vars; running in parallel (`pytest-xdist`) races.

```python
os.environ["GATEWAY_CONFIG"] = str(cfg_path)
os.environ["GATEWAY_DB_DSN"] = pg_dsn
...
os.environ["GATEWAY_SEED_CALLERS"] = "1"
```

b) `GATEWAY_SEED_CALLERS=1` is set — but `gateway/app.py` does NOT read this variable. It's either a test for a feature that doesn't exist (the gate I recommended in cr-1 §2.2), or dead config. Either way the test currently passes for the wrong reason: seeding happens unconditionally.

c) Imports `gateway.app.app` *after* setting env vars — but `app` is module-scoped and may have been imported earlier by another test (e.g., when collecting). Once imported, the lifespan-managed state binds to the env vars seen at import time. If `test_app_e2e.py` runs after a test that already imported `gateway.app`, the import is cached and the env-var changes have no effect.

**Fix:** use `monkeypatch.setenv` inside a function-scoped fixture; explicitly `importlib.reload(gateway.app)`; or factor `create_app(cfg, ...)` so the test constructs an app without env-var ceremony.

### 11.2 `test_healthz`
**Solid.**

### 11.3 `test_chat_completions_happy_path`
**Solid.** Could assert at least one row in the `requests` table after the call (the test docstring promises "attempts persisted" but no test actually checks the DB).

### 11.4 `test_chat_completions_requires_auth`
**Solid.**

### 11.5 `test_chat_completions_rejects_stream_in_v1`
**Improvable.** The docstring says step 14 "rejects stream"; status check is `== 422`. FastAPI bubbles the Pydantic `ValueError` as a 422, but a future Pydantic upgrade could change the exact status (some FastAPI configs return 400 for field validators). Loosen to `r.status_code in (400, 422)` or use `r.json()["detail"]` to assert the message contains "stream".

### 11.6 `test_daily_cap_returns_429`
**Improvable.** Loops up to 30 attempts hoping for a 429 — passes today because each mock attempt yields enough tokens. If the mock vendor's default token counts ever shrink, the loop completes without triggering 429 and the test fails confusingly.

### 11.7 `test_metrics_contains_routing_weight`
**Solid.** Verifies key metric names appear in the scrape output.

### 11.8 `test_weighted_distribution_approximates_config`
**Brittle.**

a) Counts attempts via `/metrics` at the *end* of the run — Prometheus counters are cumulative across **all prior tests in the module**, so the warmup request in `test_metrics_contains_routing_weight` and the `test_daily_cap_returns_429` loop also contribute. The 50/30/20 check is fuzzed enough (±10%) to hide this today, but adding any new test that issues requests before `test_weighted_distribution_approximates_config` will shift the ratios.

b) `±10% absolute tolerance` over N=400 trials is statistically loose. For a true 50/30/20 distribution the 99% confidence interval on each ratio is ~±6%; 10% admits a meaningful skew (e.g., 41/35/24).

c) The test does NOT clear the engine cache before running, so previous tests' breaker/observe state carries over.

**Fix options:**

## 20. Cross-cutting Suggestions

1. **Promote duplicated builders to `conftest.py`.** `_config()` is rewritten in `test_router.py`, `test_refresh.py`, `test_app_e2e.py`, and `test_build_vendors.py`, each slightly different. A `make_config(**overrides)` factory plus a `simple_caller()` helper eliminates a class of "test passes for the wrong reason" bugs caused by config drift.

2. **Reset Prometheus REGISTRY between tests.** `gateway/metrics.py` registers Counters/Gauges at import time on a module-level `REGISTRY`. Any test that scrapes `/metrics` is reading cumulative state. Add:
   ```python
   @pytest.fixture(autouse=True)
   def _reset_metrics():
       from gateway.metrics import REGISTRY
       for c in list(REGISTRY._collector_to_names.keys()):
           # Use a fresh registry per test, or clear samples.
           ...
   ```

3. **Replace SDK-attribute monkeypatching with `respx`.** `respx==0.21.1` is already in the dev deps but no test uses it. HTTP-level mocking exercises the SDK's serialization, header handling, and retry logic — closer to production.

4. **Run one concurrency test against testcontainers Redis.** fakeredis serializes commands; the breaker-probe and token-bucket atomicity claims need a real Redis to be meaningful. The e2e fixture already starts one — add a `test_probe_lock_real_redis_atomic` test there.

5. **Adopt `pytest.mark.parametrize` for error-mapping tests.** Each provider's six error-mapping tests are copy-paste — one parametrized test would catch a forgotten branch (e.g., `ContentFiltered`) at design time.

6. **Tighten "or" assertions and "≤" one-sided checks.** Tests that assert `kind in (A, B)` or `value <= upper_bound` accept too much. Pin to one expected value or use closed ranges.

7. **Add a `/metrics` snapshot test** — golden file of the Prometheus output (with timestamps stripped) so accidental label-name drift is caught at the byte level.

8. **Add a missing-vendor test path in the router.** The `vendor is None` branch at `router.py:181-185` has zero coverage.

9. **Consider hypothesis (property-based) tests** for `health_score`, `budget_score`, `effective_weight`, and `estimate_tokens` — all pure, all numerically tricky, all good candidates.

10. **Coverage gap**: there is no `pytest --cov` configuration in `pyproject.toml`. Add `[tool.coverage.run] source = ["gateway"]` and a CI check at e.g. 85% to keep the suite honest.
